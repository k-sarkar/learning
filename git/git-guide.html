Git Core Concepts

Repository (Repo)

A complete project history stored in a .git directory. Contains all commits, branches, and metadata.

~/project/.git/

Commit

A snapshot of your code at a point in time, with a unique hash, message, author, and timestamp.

abc1234d... "Fix login bug"

Branch

A lightweight pointer to a specific commit. Default branch is 'main'. Allows parallel development.

main, feature/auth, bugfix/123

HEAD

A reference pointing to the current commit. Usually points to a branch name.

HEAD → main (at commit abc1234)

Index / Staging Area

A buffer zone between your working directory and the repository. Use it to prepare commits.

git add file.txt (to stage)

Working Directory

The actual files on your computer. Modified files show up as "unstaged changes".

~/project/ (all your files)
Git Architecture: Three Zones

Local vs Remote

Local Repository

Your copy of the project on your computer (.git folder). Complete history and all operations offline.

Remote Repository

Shared copy on a server (GitHub, GitLab, etc.). Acts as the "source of truth" for team collaboration.

Local Git Operations

Setting Up Git
git config --global user.name "Your Name"
git config --global user.email "you@example.com"
Click any command to copy

Initialize & Clone

Creating a New Repository
git init
Creates a new .git folder in the current directory. Turns a normal folder into a Git repository.
Clone an Existing Repository
git clone https://github.com/user/repo.git
Downloads the entire project from remote to your local machine. Sets up local tracking branches automatically.

Checking Status

View Current Status
git status
Shows which files are modified, staged, or untracked. Essential for understanding your working state.
See Differences
git diff
Shows unstaged changes (working directory vs index). Use git diff --staged for staged changes.
View Log
git log --oneline -n 10
Shows last 10 commits in compact form. Remove -n 10 to see all commits.

Staging & Committing

Stage Changes (Add to Index)
git add file.txt
git add .
git add -p
git add file.txt: Stage specific file
git add .: Stage all changes in current directory
git add -p: Interactive staging (review changes chunk by chunk)
Create a Commit
git commit -m "Clear, descriptive message"
git commit -am "Message"
-m: Inline commit message
-a: Auto-stage tracked files (skip git add), then commit
Commit Workflow
Modify Files
→
git add
→
git commit
→
History Saved

Undoing Changes

Unstage a File
git reset HEAD file.txt
Removes file from staging area but keeps changes in working directory.
Discard Unstaged Changes
git checkout -- file.txt
git restore file.txt
Throws away changes in working directory. WARNING: This cannot be undone!
Undo Last Commit
git reset --soft HEAD~1
git reset --mixed HEAD~1
git reset --hard HEAD~1
--soft: Undo commit, keep changes staged
--mixed: Undo commit, keep changes unstaged (default)
--hard: Undo commit, discard all changes (irreversible)
⚠️ Destructive Operations: --hard, checkout --, restore can permanently delete changes. Commit often and push to remote to avoid data loss!

Remote Repository & Collaboration

Understanding Remote

origin

Default remote name. Points to the server repository where you cloned from (usually GitHub).

Tracking Branch

Local branch that tracks a remote branch (e.g., main tracks origin/main). Know what's ahead/behind.

Remote Branch

Read-only reference to remote branches. Updates only when you fetch/pull. Shown as origin/main, origin/feature, etc.

Managing Remotes

View Remotes
git remote -v
Shows all configured remotes with URLs. -v shows both fetch and push URLs.
Add/Remove Remote
git remote add origin https://github.com/user/repo.git
git remote remove origin
Add a new remote or remove an existing one.

Fetch, Pull, Push

Fetch vs Pull vs Push
Fetch (Download Without Merging)
git fetch
git fetch origin
Downloads new commits from remote. Updates remote-tracking branches (origin/main, origin/feature, etc.) but does NOT merge into your working branch. Safe to run anytime.
Pull (Fetch + Merge)
git pull
git pull --rebase
Default (merge): Fetches and merges remote changes. Creates merge commit if there are new commits.
--rebase: Fetches and rebases your commits on top of remote. Cleaner history but more complex for beginners.
Push (Upload Local Commits)
git push
git push origin main
git push --all
Default: Push current branch to configured remote tracking branch
Explicit: Specify remote and branch
--all: Push all branches
⚠️ Collaboration Tip: Always pull before pushing! If someone else pushed changes, you'll get a rejection. Pull first to merge their changes, then push yours.

Working With Others

Check What's Different
git fetch origin
git log --oneline main..origin/main
Shows commits on remote that aren't on your local main. Use this before pulling to see what you're getting.
Check Your Unpushed Work
git log origin/main..main
Shows commits on your local main that aren't on remote yet. Ready to push!

Branching Strategy

Why Branches?

Branches let multiple developers work in parallel. Isolate features, fixes, and experiments without affecting main codebase.

Main Branch

Usually called 'main' or 'master'. Stable, production-ready code. Protected on most projects (requires reviews).

Feature Branch

Short-lived branch for a specific feature. Examples: feature/user-auth, feature/dark-mode. Merged back to main when done.

Creating & Switching Branches

List Branches
git branch
git branch -a
Local branches only: git branch
All (local + remote): git branch -a
Create a New Branch
git branch feature/auth
git checkout -b feature/auth
git switch -c feature/auth
git branch: Create branch but stay on current
git checkout -b: Create and switch in one command
git switch -c: Modern syntax (Git 2.23+)
Switch Between Branches
git checkout main
git switch main
Switch to an existing branch. Must commit or stash changes first.
Branch Workflow

Merging Branches

Merge a Feature Branch Into Main
git checkout main
git pull origin main
git merge feature/auth
git push origin main
1. Switch to main
2. Get latest from remote
3. Merge feature branch into current branch
4. Push merged changes
Merge Conflicts
When you merge and both branches changed the same lines, Git can't decide automatically. You must resolve it manually.
git status
Shows files with conflicts (marked as "both modified").
Resolving Conflicts
Git marks conflicts with special markers: <<<<<<< HEAD
your changes
=======
their changes
>>>>>>> branch-name
1. Edit the file and choose which changes to keep
2. Remove the <<<, ===, and >>> markers
3. Stage and commit
Abort a Problematic Merge
git merge --abort
Cancel the merge and return to the state before merge started.

Deleting Branches

Delete Local Branch
git branch -d feature/auth
git branch -D feature/auth
-d: Safe delete (only if merged)
-D: Force delete (even if not merged)
Delete Remote Branch
git push origin --delete feature/auth
git push origin :feature/auth
Remove branch from remote server. Useful after merging to clean up.

Advanced Git Techniques

Stash: Temporarily Save Work

Stash is used when you have uncommitted changes and need to switch branches without committing. Perfect for "work-in-progress" code you're not ready to commit yet.
Save Work With Stash
git stash
git stash push -m "WIP: user authentication"
git stash: Save all unstaged changes
With message: Add a description for easy reference
List Stashed Changes
git stash list
Shows all stashes with IDs (stash@{0}, stash@{1}, etc.)
Restore Stashed Work
git stash pop
git stash apply stash@{0}
pop: Restore latest stash and remove it
apply: Restore specific stash but keep it
Delete Stashes
git stash drop stash@{0}
git stash clear
Remove a specific stash or all stashes.
Stash Workflow Example
$ git status On branch feature/auth
Modified: auth.js
Modified: login.html
$ git stash push -m "WIP: auth flow" Saved working directory clean
$ git switch main Switched to branch 'main'
$ git stash pop Restored from stash@{0}

Rebase: Rewrite History

Rebase moves your commits on top of another branch. Creates a linear history (no merge commits) but rewrites history. Use with caution on shared branches!
Interactive Rebase (Rewrite Commits)
git rebase -i HEAD~3
Opens editor to modify, squash, or reorder last 3 commits. -i means "interactive".
Rebase Branch On Main
git rebase main
Takes your feature branch commits and replays them on top of latest main. Cleaner than merge commit.
Abort Rebase
git rebase --abort
Cancel the rebase and return to original state. Use if you encounter complex conflicts.
⚠️ Golden Rule of Rebase: Never rebase commits that are already pushed to a shared remote! It rewrites history and causes chaos for teammates. Only rebase local branches.
Merge vs Rebase

Cherry-Pick: Apply Specific Commits

Apply a specific commit from another branch to current branch. Useful for backporting fixes.
Cherry-Pick a Commit
git cherry-pick abc1234
git cherry-pick abc1234..def5678
Single commit: Apply specific commit hash
Range: Apply all commits between two hashes
Handle Cherry-Pick Conflicts
git cherry-pick --continue
git cherry-pick --abort
Resolve conflicts and continue, or abort the operation.

Reflog: Recover Lost Commits

Reflog shows every change to HEAD. Use it to recover commits you thought were deleted!
View Reflog
git reflog
Shows all HEAD movements. Entries are time-based (HEAD@{0}, HEAD@{1}, etc.)
Recover a Lost Commit
git reset --hard abc1234
git reset --hard HEAD@{3}
Go back to a specific commit or reflog entry. Useful after accidental hard resets!

Useful Commands & Tricks

Undo Last Push (If Not Shared)
git revert HEAD
git push origin main --force
Creates new commit that undoes changes. Safer than reset --hard for pushed commits.
Squash Commits Into One
git rebase -i HEAD~3
Use interactive rebase, then change 'pick' to 'squash' for commits you want to combine.
Search Commit History
git log --oneline --grep="bug fix"
git log -S "function_name"
Find commits by message or by code changes. Helpful for tracking down when something changed.
See Who Changed What (Blame)
git blame file.js
Shows each line's author, commit hash, and date. Find who wrote problematic code!

Complete Git Workflows

Workflow 1: Creating & Merging a Feature

Step-by-Step: Feature Development
git checkout -b feature/user-dashboard
Create and switch to new feature branch.
echo "new code" > dashboard.js
Make your changes (edit files).
git add dashboard.js
Stage your changes.
git commit -m "feat: add user dashboard"
Commit with clear message.
git push -u origin feature/user-dashboard
Push to remote. -u sets upstream for future pushes.
git checkout main
Switch back to main.
git pull origin main
Get latest main from remote.
git merge feature/user-dashboard
Merge feature into main.
git push origin main
Push merged code to remote.
git branch -d feature/user-dashboard
Clean up local branch (optional).
git push origin --delete feature/user-dashboard
Clean up remote branch (optional).

Workflow 2: Collaborative Development (Team)

Scenario: Pulling Changes From Teammates
git fetch origin
Check for new changes without merging.
git log main..origin/main
See what your teammates pushed.
git pull origin main
Download and merge their changes into your local main.
git rebase origin/main
Alternative: Rebase your work on top of latest main (cleaner history).

Workflow 3: Emergency Fix (Hotfix)

Quick Production Fix While Working on Feature
git stash
Save your current work (feature branch).
git checkout main
Switch to main for the hotfix.
git checkout -b hotfix/critical-bug
Create hotfix branch from main.
echo "fix code" > critical.js
Make and commit the fix.
git add . && git commit -m "fix: critical production bug"
Commit the hotfix.
git push origin hotfix/critical-bug
Push hotfix branch.
git checkout main && git merge hotfix/critical-bug && git push
Merge hotfix into main and push.
git checkout feature/user-dashboard
Back to your feature branch.
git stash pop
Restore your stashed work.

Workflow 4: Rebase Before Merge Request

Clean History Before Creating Pull Request
git fetch origin
Get latest from remote.
git rebase origin/main
Replay your commits on top of latest main.
git rebase -i HEAD~5
Clean up your last 5 commits (squash, reorder, etc.).
git push origin feature/my-feature --force-with-lease
Push rebased branch. --force-with-lease is safer than --force.

Workflow 5: Syncing Your Fork

Update Your Fork With Upstream Changes
git remote add upstream https://github.com/original/repo.git
Add reference to original repo (one-time setup).
git fetch upstream
Get latest from original repo.
git checkout main
Switch to main branch.
git rebase upstream/main
Update your main with original repo's main.
git push origin main --force-with-lease
Push updated main to your fork.

Checklist: Before Your First Commit

  • Configure user name and email: git config --global user.name
  • Understand the three zones: Working → Index → Repository
  • Write clear commit messages (imperative mood: "Add feature" not "Added feature")
  • Commit frequently with logical, atomic changes
  • Always pull before pushing to avoid conflicts
  • Never force-push to shared branches
  • Use branches for features, fixes, and experiments
  • Review what you're committing with git diff before staging
